# 冲突解决工作台与同步中心 UI 优化计划

## 背景

当前 WebDAV 同步插件已经具备冲突检测、Blob 绑定的本地/远程选择、同步日志、待同步列表、能力报告等基础能力。但现有 UI 把状态、能力、冲突、待同步、历史和日志全部纵向堆在同一个同步中心弹窗中。

当冲突文件较多时，用户需要点击底部状态栏打开同步中心，再在弹窗中滚动查找冲突项。Markdown 冲突目前只展示 BASE / LOCAL / REMOTE 文本块，没有类似 IDEA / JetBrains 的行级差异高亮，也没有清晰的多文件逐个处理流程。

本计划目标是把“查看同步状态”和“处理冲突”拆成两个清晰体验：同步中心负责概览和诊断，冲突解决工作台负责逐个文件解决冲突。

## 当前实施状态（2026-07-16）

当前 `0.9.0` 已完成以下内容：

1. 同步中心已改为概览、待同步、历史、日志和能力分区。
2. 冲突状态下，底部状态栏点击会打开独立的冲突解决工作台。
3. 工作台支持多文件切换、全部/未处理/已处理筛选、本地或远程版本选择，以及全部解决后继续同步的门禁。
4. Markdown 冲突完整的 BASE / LOCAL / REMOTE 文本只保留在内存中；复制诊断仍然不会包含正文。
5. 已实现带行号的三栏差异高亮和同步滚动；超大文本自动降级为简化视图，避免 UI 卡顿。

仍待真实 Obsidian GUI 验证的项目：明暗主题的实际可读性、窄窗口布局、完整冲突处理后再次同步的交互闭环。

## 产品目标

1. 底部状态栏在冲突状态下点击后直接进入冲突解决工作台。
2. 冲突解决工作台支持多个文件依次处理。
3. Markdown 冲突展示类似 IDEA 的三方对比视图，并对差异行进行高亮。
4. 每个冲突文件必须明确选择处理方式；全部冲突解决完成后才能继续同步。
5. 同步中心 UI 改为更清晰的信息控制台，避免所有信息同时铺开。
6. 保留现有安全机制：冲突选择必须绑定 base/local/remote Blob，内容变化后旧选择自动失效。

## 非目标

第一阶段不做以下功能：

1. 行级接受本地块或远程块。
2. 手动编辑最终合并结果。
3. 历史版本浏览。
4. 二进制文件内容预览。
5. 远程文件导出或另存为。

这些能力可以作为后续增强，避免第一版过大、风险过高。

## 目标体验

### 状态栏入口

底部状态栏点击行为按当前状态分流：

```text
存在冲突     → 打开冲突解决工作台
其他状态     → 打开同步中心
```

命令面板保留“打开同步中心”，并新增“处理同步冲突”。同步中心内部在存在冲突时也显示“处理冲突”主按钮。

### 同步中心

同步中心从单页纵向内容改为分区控制台：

```text
Header：当前状态 / 主操作 / 复制诊断
Tabs：概览 / 待同步 / 历史 / 日志 / 能力
Content：当前 Tab 内容
```

概览页只展示最重要的信息：

```text
状态
待同步数量
冲突数量
最近一次同步结果
当前推荐动作
```

待同步页展示本地更改队列，按 create / modify / delete / rename 分组或标记。

历史页展示最近同步记录，失败和冲突高亮。

日志页支持按 info / warn / error 过滤，默认不展示 debug。

能力页展示 WebDAV 能力报告，警告置顶，技术细节可折叠。

### 冲突解决工作台

工作台采用三栏主布局：

```text
左侧：冲突文件列表
右侧：当前文件对比视图
底部：处理进度与操作按钮
```

左侧列表显示：

```text
欢迎.md                  未处理
日记/2026-07-16.md        已选择本地
assets/logo.png           二进制冲突
```

支持筛选：

```text
全部 / 未处理 / 已处理
```

右侧 Markdown 文本冲突使用三方只读对比：

```text
基础版本        本地版本        远程版本
BASE            LOCAL           REMOTE
```

每一栏需要具备：

1. 行号。
2. 等宽字体。
3. 差异行高亮。
4. 当前冲突块强调。
5. 三栏同步滚动。
6. 长行不撑破布局。

底部操作：

```text
上一个
下一个
使用本地版本
使用远程版本
完成并继续同步
```

当还有未处理冲突时，“完成并继续同步”禁用，并显示：

```text
还有 N 个冲突未处理
```

所有可解决冲突都有选择后，按钮启用。点击后调用现有冲突选择结果重新执行同步。

## 视觉设计原则

1. 使用 Obsidian 原生主题变量，适配明暗主题。
2. 不使用大面积装饰色，保持工具型界面克制、密集、可扫描。
3. 冲突列表和对比视图要有稳定尺寸，切换文件时布局不跳动。
4. 按钮只承担明确命令：使用本地、使用远程、上一个、下一个、完成并继续同步。
5. 技术诊断信息默认折叠或放入专门 Tab，不干扰普通同步流程。

建议高亮语义：

```text
新增       → 绿色淡背景
删除       → 红色淡背景
修改       → 黄色/橙色淡背景
普通上下文 → 默认背景
已选择文件 → 使用主题 accent 边线或背景
```

## 技术设计

### 新增 UI 模块

建议新增文件：

```text
src/ui/conflict-resolver-modal.ts
src/ui/three-way-diff-view.ts
src/ui/diff-lines.ts
```

职责划分：

```text
conflict-resolver-modal.ts  负责冲突工作台布局、文件选择、处理进度和继续同步
three-way-diff-view.ts      负责三方对比 DOM 渲染和同步滚动
diff-lines.ts               负责把 base/local/remote 文本转换为可渲染的行级差异模型
```

同步中心继续保留在：

```text
src/ui/sync-center-modal.ts
```

但需要重构为 Tab 式信息架构。

### 数据结构调整

当前冲突快照大致为：

```ts
{
  path: string;
  action: string;
  canResolve: boolean;
  choice?: ConflictChoice;
  details?: { base: string; local: string; remote: string };
}
```

建议扩展为：

```ts
interface ConflictViewModel {
  path: string;
  action: string;
  canResolve: boolean;
  choice?: ConflictChoice;
  baseBlob: string | null;
  localBlob: string | null;
  remoteBlob: string | null;
  versions?: {
    base: string;
    local: string;
    remote: string;
  };
}
```

注意：

1. `versions` 只放内存，不持久化到 `data.json`。
2. 复制诊断信息时继续省略正文，只保留脱敏后的结构信息。
3. `chooseConflict()` 继续绑定 `baseBlob / localBlob / remoteBlob`，防止陈旧选择误用。

### Diff 模型

第一版使用只读行级 diff，不做行级合并。

建议输出模型：

```ts
type DiffLineKind = "context" | "added" | "removed" | "changed" | "conflict";

interface DiffRow {
  base?: DiffCell;
  local?: DiffCell;
  remote?: DiffCell;
}

interface DiffCell {
  lineNumber: number | null;
  text: string;
  kind: DiffLineKind;
}
```

实现策略：

1. 先按行拆分 base/local/remote。
2. 对 base→local、base→remote 分别做 LCS 行 diff。
3. 合并两个 diff 结果，标记三栏中的新增、删除、修改和冲突区域。
4. 对长文件做上下文折叠可以放到第二阶段；第一版先完整展示。

### 同步流程

冲突解决工作台完成时：

```text
检查所有 canResolve 冲突是否都有 choice
否 → 禁用继续同步
是 → runManualSync()
```

同步引擎沿用现有逻辑：

```text
hasValidResolution(item, resolution)
```

如果文件内容变化导致 Blob 不匹配，选择自动失效，并重新进入冲突状态。

## 分阶段实施

### 阶段 1：同步中心信息架构重构

目标：让底部状态弹窗不再拥挤。

任务：

1. 将 `SyncCenterModal` 改为 Header + Tabs + Content。
2. 新增概览、待同步、历史、日志、能力五个 Tab。
3. 冲突内容不再在同步中心内展开大段正文。
4. 存在冲突时，概览页显示“处理冲突”主按钮。
5. 保留复制诊断按钮。

验收：

1. 普通状态下点击底部状态打开同步中心。
2. 同步中心默认只展示概览，不铺满日志和能力细节。
3. 待同步、历史、日志、能力均可切换查看。
4. 复制诊断仍可用，且不包含笔记正文。

### 阶段 2：冲突解决工作台基础流程

目标：多个冲突文件可以逐个处理，全部处理完才能继续同步。

任务：

1. 新增 `ConflictResolverModal`。
2. 状态栏在 conflict 状态下点击打开冲突解决工作台。
3. 左侧展示冲突文件列表和处理状态。
4. 右侧先展示现有 BASE / LOCAL / REMOTE 三栏文本。
5. 每个文件支持“使用本地版本 / 使用远程版本”。
6. 选择后更新左侧状态，并可自动跳到下一个未处理文件。
7. 全部解决前禁用“完成并继续同步”。
8. 全部解决后点击按钮重新执行同步。

验收：

1. 冲突状态下点击底部状态直接进入工作台。
2. 多个冲突可以依次处理。
3. 未全部处理时不能继续同步。
4. 全部处理后能触发现有同步重试。
5. Blob 绑定失效时旧选择不会被误用。

### 阶段 3：IDEA 风格三方 Diff 视图

目标：Markdown 冲突可读性接近 IDE 对比工具。

任务：

1. 新增 `diff-lines.ts` 生成行级三方 diff 模型。
2. 新增 `three-way-diff-view.ts` 渲染三栏行号和高亮。
3. 实现三栏同步滚动。
4. 为新增、删除、修改、冲突块增加样式。
5. 移动端或窄窗口改为纵向堆叠，但保持文件列表和操作按钮可用。

验收：

1. Markdown 冲突显示三栏行号。
2. 差异行有明确高亮。
3. 长文本区域滚动稳定。
4. 切换冲突文件时布局不跳动。
5. 明暗主题下高亮仍可读。

### 阶段 4：打磨与测试

目标：确保 UI 流程稳定，不破坏同步安全性。

任务：

1. 为设置、冲突选择、同步中心快照补充单元测试。
2. 为 diff 模型补充文本场景测试。
3. 构建 Obsidian 测试 Vault 并手动验证真实弹窗。
4. 检查复制诊断不泄露正文。
5. 检查移动端窄宽度布局。

验收：

1. `npm run check` 通过。
2. `npm run build` 通过。
3. 冲突工作台在真实 Obsidian 中可打开、选择、继续同步。
4. 同步中心信息层级清晰，不再默认铺开冲突正文和大量日志。

## 版本建议

最初建议按以下版本拆分：

```text
0.9.0   同步中心 Tab 化 + 冲突入口分流
0.10.0  冲突解决工作台 + 多文件逐个处理 + 全部解决后继续同步
0.11.0  IDEA 风格三方 Diff 高亮 + 同步滚动
```

实际实现已将上述主流程和基础行级高亮合并到 `0.9.0`。后续版本应重点考虑行级接受单侧改动、手动编辑合并结果，以及真实 Obsidian GUI 验证后的视觉细节调整。

## 风险与处理

### 风险 1：笔记正文进入持久化或诊断

处理：冲突正文只保留在运行时内存；`copyDiagnostics()` 继续使用脱敏结构，不复制 `versions` 内容。

### 风险 2：用户处理冲突期间文件变化

处理：继续使用 `baseBlob / localBlob / remoteBlob` 绑定选择；同步时不匹配则旧选择失效并重新展示冲突。

### 风险 3：Diff 计算复杂导致 UI 卡顿

处理：第一版完整渲染，后续如遇超大文件再加上下文折叠或大文件降级提示。

### 风险 4：同步中心重构影响已有诊断能力

处理：先保留所有现有信息，只改变信息架构；能力报告、日志、历史和待同步列表仍可访问。

## 推荐下一步

优先实现阶段 1 和阶段 2：

```text
先让同步中心变清楚
再让冲突处理形成闭环
最后打磨 IDEA 风格 diff 视觉
```

这样每个阶段都有独立可用价值，也能避免把视觉细节和同步安全逻辑绑在一次大改里。
